## 1. 문제

일반 텍스트 마커를 줌인했을 때 같은 줌 구간에서도 점 마커가 말풍선 마커로
바뀌지 않는 경우가 있다. 사용자는 iOS 실기기에서 빠르게 줌인할 때 이 현상을
관찰했다.

현재 전환 기준은 다음과 같다.

- 점 -> 말풍선: 줌 `17.3` 이상
- 말풍선 -> 점: 줌 `16.8` 미만
- 바텀 카드가 열린 동안은 말풍선을 점으로 억제
- 카드가 닫히면 현재 줌에 맞는 표현으로 복원

이번 작업은 전환 누락을 복구하는 정확성 수정이다. 줌 제스처 도중 점과
말풍선을 교차 페이드하는 크로스페이드는 이번 범위에 포함하지 않는다.

## 2. 확인된 원인

`7efdff4 feat: 지도 마커 재조회 정책 개선`에서 점·말풍선 표현 변경을 API
재조회 조건에서 제거했다. 이 방향은 맞지만 기존 `_onCameraIdle()`의
`_peekTextCardMode()` 판단과 표현 갱신 트리거까지 함께 제거됐다.

현재 `_resolveTextCardMode()`는 `_loadMapMarkers()`가 실행될 때만 호출된다.
따라서 다음 조건이 겹치면 카메라가 멈춰도 로컬 표현 갱신이 없다.

- 현재 API 줌 단계가 마지막 성공 조회와 같음
- 지도 이동이 화면 짧은 변의 30% 미만
- 30초 freshness 갱신 시점이 아님

`_onCameraIdle()`은 `MapReloadReason.none`으로 종료하고 점 마커는 줌 `17.3`을
넘어도 그대로 남을 수 있다. 빠른 줌은 직접 원인이 아니라 API 줌 단계 변경이나
재조회 발생 여부에 따라 결과가 달라져 문제를 더 쉽게 드러내는 재현 조건이다.

현재 계약 테스트는 표현 변경 때문에 API를 재조회하지 않는지만 검증한다.
API 없이 로컬 표현을 반드시 갱신한다는 계약이 빠져 있다.

## 3. 선택한 해결 방식

`_onCameraIdle()`에서 공간 재조회 사유를 판단하기 전에 현재 줌을 기준으로
텍스트 마커 표현을 로컬에서 동기화한다.

핵심 규칙은 다음과 같다.

1. 표현 전환과 API 재조회를 독립시킨다.
2. 기존 `_postGroups`, `_markerRefs`, 마커 캐시만 사용한다.
3. 기존 `_flipTextMarkerModeInPlace()`를 재사용해 `setIcon`, `setSize`,
   `setCaption`, `setAlpha`만 갱신한다.
4. 표현 동기화 경로에서 `_loadMapMarkers()`, `addOverlay`, `deleteOverlay`를
   호출하지 않는다.
5. 점·말풍선 교체 중 마커를 숨기는 페이드인은 사용하지 않는다.

### 3.1 표현 대상

말풍선으로 바꿀 수 있는 대상은 기존 규칙을 유지한다.

- 사진이 없는 텍스트 글
- 단일 글 그룹
- 현재 줌의 화면 dp 기준으로 비밀집 지역
- 바텀 카드·스택 상호작용으로 억제된 상태가 아님

밀집 판정은 `_loadMapMarkersInternal()`과 로컬 동기화가 동일한 헬퍼를 사용한다.
줌인으로 화면상 거리가 벌어지면 새 API 응답이 없어도 현재 표시 데이터로
밀집 여부를 다시 계산한다.

### 3.2 빠른 줌 경합 방어

말풍선 비트맵 생성은 비동기이므로 줌인 직후 줌아웃하면 이전 요청이 나중에
완료될 수 있다. 표현 동기화 시도마다 증가하는 세대 번호를 두고 아이콘 적용
직전에 최신 세대인지 확인한다.

- 현재 세대와 다른 비동기 결과는 폐기
- 재조회 중 마커가 교체되면 참조 동일성 검증으로 폐기
- 네이티브에서 제거된 마커는 `isAdded` 검증으로 폐기
- 개별 마커 실패는 기존 표현을 유지하고 나머지 동기화는 계속

표현 세대는 API 재조회 세대와 분리한다. 데이터 신선도와 표현 신선도는 서로
다른 관심사이기 때문이다.

### 3.3 전환 깜빡임 후속 수정

현재 `_flipTextMarkerModeInPlace()`는 기존 마커의 alpha를 즉시 `0`으로 내린 뒤
아이콘을 교체하고, 33ms 주기의 타이머로 165ms 동안 새 표현만 페이드인한다.
두 표현을 동시에 그리는 크로스페이드가 아니므로 일반 점 마커가 사라진 다음
말풍선이 나타나며, 최소 한 프레임의 빈 구간이 깜빡임으로 보인다.

이번 후속 수정은 안전성을 우선해 다음과 같이 처리한다.

- `_fadeInMarker()`와 텍스트 전환용 alpha 타이머를 제거한다.
- 전환 직전에 기존 마커의 alpha를 `1`로 확정한다.
- 같은 마커에 `setIcon`, `setSize`, `setCaption`을 즉시 적용한다.
- 전환 과정에서 마커를 숨기거나 별도 오버레이를 추가하지 않는다.
- iOS 클러스터 leaf 수렴 보정과 세대·선택·억제 가드는 그대로 유지한다.

결과적으로 전환은 한 프레임의 직접 교체가 된다. 진짜 크로스페이드보다 형태
변화는 선명하지만, 빈 프레임과 네이티브 오버레이 추가 위험 없이 항상 점 또는
말풍선 중 하나가 보인다. 사전 등록형 이중 마커 크로스페이드는 별도 후속 작업으로
유지한다.

### 3.4 (2026-07-18) in-place 전환 폐기 — payload 재생성으로 전면 교체

3.3까지 적용 후에도 iOS에서 점↔말풍선이 **계속 깜빡이고** 표현이 줌과
어긋나는 현상이 재발했다. 근본 원인:

지도 마커는 전부 `NClusterableMarker`이고, 네이티브 클러스터러는
리클러스터링(카메라 이동·줌·오버레이 추가)마다 **최초 add payload의
아이콘·크기·캡션으로 leaf를 다시 그린다** (flutter_naver_map#276 유사, iOS).
따라서 in-place `setIcon` 전환은 본질적으로 일시적이며, 다음 리클러스터링에서
반드시 예전 표현으로 되돌아온다. 350ms 수렴 타이머는 이 되돌림을 매번
"되돌림 → 재적용" 깜빡임 사이클로 만들었고, `nowCameraPosition`(실험적)의
stale 줌이 히스테리시스 상태를 오염시켜 줌 이상 증상을 더했다.

**새 원칙: 표현의 유일한 진실은 add payload다.**

- 점↔말풍선 전환 = 마커 재생성: 새 아이콘을 전부 준비한 뒤
  `deleteOverlay` → `addOverlayAll` 일괄 교체 (`_recreateSingleTextMarkers`).
- ⚠️ delete 는 add 전에 **반드시 완료를 기다린다**: 플러그인 `deleteOverlay`
  는 네이티브 ack 이후에 Dart 탭 핸들러를 제거하고 `addOverlayAll` 은 호출
  즉시 등록하므로, await 없이 이어 부르면 늦은 delete ack 가 같은 id 의 새
  핸들러를 지워 탭이 영구히 무시된다 (overlay_controller_impl Null check).
- 마커 구성은 `_buildSingleTextClusterableMarker`가 담당하며 updateMarkers의
  신규 마커 분기와 표현 계약(tags/캡션/zIndex/탭)이 같아야 한다.
- 제거: `_flipTextMarkerModeInPlace`, 수렴 타이머 일체
  (`_scheduleTextMarkerPresentationConvergence`, `_reapplyCached...`),
  `nowCameraPosition` 의존, add 후 `force` 억제 재적용, restoreIds 북키핑.
- 동기화 트리거: ① camera idle ② 재조회 성공 직후 수렴 패스
  (`_syncTextMarkerPresentationWithCurrentCamera`, `_isLoadingMarkers` 해제
  후) ③ 카드 닫힘 복원. 셋 다 같은 함수로 수렴하며 불일치가 없으면 no-op.
- 경합 가드: 표현 세대(latest-wins) + 이동 세대(stale 카메라 폐기) +
  재조회 중 교체 금지(`_isLoadingMarkers`). 아이콘 실패·세대 불일치 시
  기존 마커 유지 — 표현 실패로 마커가 사라지지 않는다.
- 억제(카드 열림) 중에는 목표가 점인 마커만 점 payload로 재확정하고,
  말풍선 복원은 닫힘 시점이 담당한다. 선택(하이라이트) 마커는 열림 중
  재생성하지 않는다.

알려진 한계(후속): 카드 열림 중 이동으로 리클러스터링이 일어나면 억제
in-place 점이 말풍선 payload로 잠깐 되돌아올 수 있다. 억제도 payload
수준으로 옮기는 것은 별도 작업.

## 4. 처리 흐름

1. `onCameraIdle` 발생
2. 현재 카메라 줌 확인
3. `17.3/16.8` 히스테리시스로 전역 표현 모드 결정
4. 현재 `_postGroups`에서 단일·비밀집 텍스트 마커 선별
5. 현재 표현과 목표 표현이 다른 마커만 in-place 갱신
6. 별도로 기존 `MapReloadPolicy`를 사용해 API 재조회 사유 판단

API 재조회 사유가 `none`이어도 5번까지의 표현 동기화는 이미 수행된다.

## 5. 오류 처리

- 말풍선 아이콘 생성 실패: 기존 점 마커 유지
- 점 아이콘 생성 실패: 기존 말풍선 마커 유지
- 세대 불일치: 오류가 아닌 stale 결과로 폐기
- 개별 마커 실패: 나머지 마커 처리 계속
- 표현 실패 때문에 API 재조회나 오버레이 재생성을 수행하지 않음

## 6. 테스트

### 6.1 자동 테스트

- 점 상태에서 `17.3` 이상이면 말풍선 모드
- 말풍선 상태에서 `16.8` 이상이면 말풍선 유지
- 말풍선 상태에서 `16.8` 미만이면 점 모드
- `_onCameraIdle()`은 API 재조회 이유가 `none`이어도 로컬 표현 동기화 호출
- 로컬 동기화는 `_loadMapMarkers`, `addOverlay`, `deleteOverlay`를 호출하지 않음
- 최신 표현 세대만 아이콘 적용 가능
- 카드가 열린 동안 점으로 억제하고 닫으면 현재 줌으로 복원
- 점·말풍선 전환은 `setAlpha(0)` 또는 타이머 페이드인을 사용하지 않음
- alpha를 `1`로 확정한 뒤 캐시 아이콘·크기·캡션을 직접 교체
- 기존 지도 재조회 정책 테스트 전체 유지

### 6.2 iOS 실기기

1. 단일 텍스트 마커 위치에서 `16.5 -> 17.5`를 느리게 줌인
2. 같은 경로를 빠르게 30회 반복
3. `17.5 -> 16.5`를 빠르게 반복해 점 복귀 확인
4. `17.3` 주변에서 빠르게 줌 방향 반전
5. 같은 API 줌 단계에서 API 재조회 없이 전환 확인
6. API 응답 대기 중 줌 방향 반전
7. 바텀 카드를 열고 닫은 뒤 현재 줌 표현 복원 확인

성공 기준은 표현이 현재 줌과 히스테리시스에 항상 일치하고 API 재조회 로그,
`overlay can't found`, 프리즈, 탭 유실이 없는 것이다.

## 7. 범위 밖

- 줌 제스처 도중 점·말풍선 크로스페이드
- 사전 등록형 점·말풍선 이중 마커
- 마커 선택 시 줌인·바텀 카드 안무 조정
- 백엔드 `/board/map/post` 조회 정책 변경
- `flutter_naver_map` 포크나 네이티브 코드 변경
- 내 지도 화면 변경

3. 기존 줌 연동 크로스페이드 설계의 iOS 안전성 실험 진행

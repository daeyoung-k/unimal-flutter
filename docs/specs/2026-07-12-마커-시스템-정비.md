# 마커 시스템 정비 — 원형 복귀 · 변형 규칙 · 클러스터 개선

> 작성일: 2026-07-12 · 상태: 1단계 구현 완료, 클러스터 개선안 결정 대기
> 피그마: [작업영역 › 마커 클러스터 UX 개선 + 마커 변형 시트](https://www.figma.com/design/zL0UifobuW5936gNE5NmZq/%EC%9E%91%EC%97%85%EC%98%81%EC%97%AD?node-id=2-80)

## 1. 배경

- 티어드롭 핀 프레임(bae1eec)을 시도했으나 **원형 커스텀 마커로 복귀**하기로 결정.
  - 원형이 사진 콘텐츠와 어울리고, 방향성 문서의 "카테고리 색상 테두리" 계획과 정합.
  - "POI와 구분 안 됨" 문제의 해법은 모양 변경이 아니라 **링 + 크기 위계**로 정리.
- 같은 좌표 글 흩뿌리기(jitter)로 인한 **마커 겹침·캡션 충돌·텍스트 카드 가림** 문제 확인
  (강남역 밀집 지역 스크린샷 기준). 개선안 3종을 피그마에 시안화.

## 2. 구현 완료 (2026-07-12)

- 사진 마커 **원형 복귀** — 흰 링 8px + 원형 클립. 핀 시절 개선점인 중앙 정사각 크롭 유지.
  anchor는 기존과 동일한 기본값(하단 중앙) — 원 중심 좌표 지정이 필요하면 `NPoint(0.5, 0.5)` 검토.
- 마커 비트맵 합성을 `lib/screens/map/marker/marker_image_factory.dart`로 **통합**
  (구 `service/image/image_service.dart` 해체). 하드코딩 색상 → `AppColors.light` 토큰 참조.
- 내지도 도트 마커를 `map/marker/story_dot_marker.dart`로 이동 — 마커 관련 코드는 전부 `map/marker/`.
- **선택 상태 강조**: z-index 부스트 + 사진 마커 `kSelectedMarkerScale`(x1.18) 확대.
  링 재합성 방식은 iOS 마커 이미지 캐시 경합 크래시 이력으로 배제 — setSize는 비트맵 재생성이 없다.
- 데드코드 제거: 구글맵 데모(`map.dart`) + `google_maps_flutter` 의존성, `marker_preview.dart`,
  `text_marker_demo.dart`(+라우트), iOS `GMSServices`/`import GoogleMaps`, Android `geo.API_KEY` meta-data.

## 3. 마커 변형 규칙 (피그마 "마커 변형 시트")

모양은 **원형 하나로 고정**. 의미 구분은 링 · 크기 · 뱃지로만 한다.

| 축 | 규칙 |
| --- | --- |
| 링 컬러 | 기본 화이트 4px. 내 글 `primary` · 새 글(24h) `accent` · 카테고리 색 (추후) |
| 링 우선순위 | 마커당 링 1개 — 카테고리 > 내 글 > 새 글. **한 화면에 색 링 2종 이상 남발 금지** |
| 크기 위계 | score 기반 3~4단계: 42 / 50(기본) / 58 / 66px(핫플, 캡션 우선권) |
| 선택 상태 | x1.18 확대 + z-index 부스트 (구현 완료) |
| 텍스트 글 | 말풍선 점(줌아웃) ↔ 카드(줌 17.3+). 밀집 지역에선 카드 대신 점 고정 |

1차 적용 권장: **내 글 primary 링** 하나만 먼저 (내지도 소유감과 직결). 나머지는 효과 보고 추가.

## 4. 클러스터(같은 자리 다중 글) 개선안 — 결정 대기

| 안 | 내용 | 비용 | 비고 |
| --- | --- | --- | --- |
| C안 (퀵윈) | 캡션은 대표(최고 score) 1개만 · 흩뿌림 간격 상향 · 밀집 지역 텍스트 글은 점 고정 | 반나절 | 급한 불 끄기 |
| A안 (본 구현) | 같은 자리 글은 줌인에도 풀지 않고 **스택 마커**(대표+겹침+N) 유지, 탭 → 하단 카드 스트립 스와이프 | 중 | 기존 bottom_card 재사용. jitter 제거 |
| B안 (보류) | spiderfy — 탭 시 부채꼴 펼침 + 점선 다리 | 대 | 예쁘지만 1인 개발 대비 효용 낮음 |

권장 순서: **C안 → A안**. A안이 "마커는 관리 도구처럼 보이면 안 된다"(방향성 §11)와 부합 —
스토리가 쌓인 자리는 마커 5개가 아니라 "명소 하나"로 보이는 게 스토맵답다.

## 5. 오픈 퀘스천

- 사진 마커 anchor: 기본(하단 중앙) 유지 vs 원 중심(0.5, 0.5) — 실기기 확인 후 결정
- score 크기 위계의 구간 경계값 (서버 score 분포 확인 필요)
- 카테고리 링 도입 시점 (카테고리 기능 자체가 방향성 문서 기준 후순위)

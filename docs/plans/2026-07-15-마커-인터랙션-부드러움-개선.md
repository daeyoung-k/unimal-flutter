# 마커 인터랙션 부드러움 개선 (2026-07-15)

## 배경

마커 관련 인터랙션 전반이 "뚝뚝 끊기는" 느낌. 비교 대상은 당근마켓 지도 —
줌 도중 말풍선이 자연스럽게 나타나고/사라지고/옮겨진다.
결론: **Flutter 한계가 아니라 구현 구조 문제.** flutter_naver_map 1.4.4 는
`setAlpha` / `setMinZoom·setMaxZoom` / `setPosition` / `setSize` 를 모두
지원하므로 애니메이션 재료는 충분하다.

## 진단 (map_naver.dart 기준)

| # | 증상 | 원인 |
|---|------|------|
| 1 | 마커 탭 → 카메라 반응 지연 | `_moveCameraToMarker` 가 애니메이션 시작 전 `getCameraPosition` + `screenLocationToLatLng` 왕복 2회 await |
| 2 | 모든 이동이 출렁거림 | 거리 무관 일괄 `fly` 600ms — 근거리에선 줌아웃→줌인 곡선이 어색 |
| 3 | 스택 펼침이 한참 멈췄다 뿅 | 카메라 비행 600ms 완료 대기 → 아이콘 **직렬** 생성(썸네일 개별 네트워크 fetch) → 전부 끝나야 addOverlayAll. `_markerBytesCache` 미활용 |
| 4 | 줌인/줌아웃 시 점↔말풍선 전환 "뚝" | 전환이 idle → 디바운스 → **API 재조회** → 아이콘 재생성 → 마커 교체에 묶여 있음. 제스처 중엔 아무 일도 없다가 한꺼번에 교체 |
| 5 | 말풍선 억제/복원 순간 점프 | setIcon 즉시 스왑 (프레임 단위 교체) |
| 6 | 포커스 크기 변화 점프 | setSize 1회 호출 — 트윈 없음 |

당근이 부드러운 이유: 표현 전환(말풍선 표시/숨김)을 **네이티브 SDK 에
위임**(overlay min/max zoom)해서 줌 제스처 도중 프레임 단위로 전환되기 때문.
API 통신·Dart 왕복과 무관.

## 설계 — 3 단계

### Phase 1 — 즉효·저위험 (카메라 + 펼침 파이프라인 + 크기 트윈)

**1-a. `_moveCameraToMarker` 왕복 제거**
- `screenLocationToLatLng` 를 수학 계산으로 대체:
  마커를 화면 세로 22% 에 두려면 카메라 중심은 마커보다 `(0.5-0.22)*height` dp
  남쪽. `getMeterPerDpAtLatitude(lat, zoom: targetZoom)` 은 **동기 순수 함수**
  → `dLat = heightDp*0.28 * meterPerDp / 111320`, `dLng = 0`.
- 남는 await 는 `getCameraPosition` 1회 (zoom 미지정 시). 탭→카메라 시작
  레이턴시 절반 이하.

**1-b. 거리 기반 애니메이션 선택**
- 이동 거리(현재 target ↔ 목표)가 화면 높이의 ~60% 이내면
  `easing` 300ms, 그 이상이면 `fly` 600ms.
- 펼침 마커 간 이동(keepStackFan)은 항상 easing — 마이크로 이동.

**1-c. 스택 펼침 병렬화**
- 반지름 계산에 settled 카메라 불필요 — `fanZoom` 으로
  `getMeterPerDpAtLatitude` 직접 계산 (동기).
- 아이콘 생성을 `Future.wait` 병렬 + `_markerBytesCache` 재사용
  (이미 그려진 링 썸네일 재다운로드 금지).
- 카메라 비행과 아이콘 준비를 **병렬** 진행, 둘 다 끝나는 시점에
  addOverlayAll → 체감 대기 = max(비행 600ms, 아이콘) ≈ 비행 시간.
- 기존 seq 경합 가드는 각 await 지점에 유지.

**1-d. 포커스 크기 트윈**
- 선택/해제 시 setSize 를 150ms / ~10 스텝 트윈 (periodic timer).
- 마커 1개라 채널 부담 없음. 진행 중 트윈은 새 트윈 시작 시 cancel.

### Phase 2 — 텍스트 마커 이중화 (줌 전환 네이티브 위임, 당근 방식)

- 텍스트 글마다 마커 2개를 **동시에** 추가:
  - **점 마커** (기존 NClusterableMarker) — `maxZoom = 전환 줌`
  - **말풍선 마커** (일반 NMarker, 클러스터 비대상) — `minZoom = 전환 줌`
- 네이티브 SDK 가 줌 제스처 **도중** 즉시 전환 → Dart 왕복 0, 재조회와 무관.
- `_peekTextCardMode` 히스테리시스는 재조회 판단용으로만 유지, 표현 전환
  로직(`_flipTextMarkerModeInPlace`)은 제거 방향.
- 말풍선 억제(카드 열림)는 말풍선 마커 `setIsVisible(false)` 또는
  `maxZoom` 조작으로 단순화 — setIcon 스왑 코드 대폭 삭제 가능.
- **검증 필요 (구현 중 확인)**:
  - 클러스터러블 점 마커에 maxZoom 이 정상 적용되는지 (클러스터링 파이프라인
    경유 시 min/max zoom 유지 여부)
  - 말풍선 마커 anchor 조정 — 점 위에 카드가 뜨는 기하가 피그마와 일치하는지
  - 탭 리스너 이중 등록 (점/말풍선 각각)
  - 오버레이 수 2배 증가의 성능 영향 (텍스트 글 한정이라 미미 예상)

### Phase 3 — 마감 폴리시

- **펼침 애니메이션**: 팬 마커를 중심 좌표·alpha 0 으로 add →
  150~200ms 동안 `setPosition`(중심→목표) + `setAlpha`(0→1) 트윈.
  다리 폴리라인은 펼침 완료 후 추가. 접기는 역방향 (또는 즉시 제거).
- **말풍선 크로스페이드**: 카드 열림/닫힘 시 말풍선 `setAlpha` 1→0 / 0→1
  120ms 트윈 (Phase 2 이중 마커 구조 전제).
- 트윈 유틸 공용화: `_animateOverlay(steps, duration, onStep)` 헬퍼 1개로
  크기·위치·알파 공유.

## 성능 위생 (전 단계 공통)

- 트윈 스텝은 30fps(33ms 간격) 기준 — 채널 왕복 부담과 부드러움의 균형.
- 다수 마커 동시 트윈 금지 (펼침 ≤ 8개 한정, 그 외 단일 마커).
- debugPrint 남발 구간은 kDebugMode 가드 유지.

## 진행 방식 (2026-07-15 확정)

**한 번에 하나씩.** Phase 1 일괄 적용을 1차 시도했다가 롤백했다 —
크기 트윈(1-d)이 말풍선 복원 경로의 setSize 와 경합해 카드가 점 크기로
찌그러졌다. 마커 크기를 만지는 경로가 이미 3갈래(선택/억제/재조회)라,
여러 항목을 동시에 넣으면 상호작용 검증이 안 된다.
→ **항목 단위로 요청받아 하나 적용 → 실기기 확인 → 커밋** 순으로 진행한다.

- **고스트 크로스페이드 롤백 (2026-07-16)**: 점↔말풍선 전환에 이전 표현
  고스트 마커를 깔아 진짜 크로스페이드를 시도 → 재조회(클러스터 파이프라인
  활동) 중 고스트 `addOverlay` 시점에 앱 프리즈 재현. 유력 원인: 네이버
  클러스터러의 내부 `dispatch_sync` 와 메인 스레드 오버레이 추가의 상호
  대기 (1차 iOS 크래시 리포트 Thread 39 참고). **재조회 창에서 단발
  addOverlay 금지** — 오버레이 추가는 addOverlayAll 배치로, 그 외 전환
  연출은 기존 오버레이의 속성(alpha/position)만으로 할 것.

- 1-a/1-b/1-c 는 마커 **크기를 건드리지 않아** 저위험 — 재작업 1순위.

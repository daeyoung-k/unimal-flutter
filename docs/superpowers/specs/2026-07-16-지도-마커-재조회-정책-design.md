# 지도 마커 재조회 정책과 실시간성 설계

- 작성일: 2026-07-16
- 상태: 구현 완료, 실기기 검증 대기
- 대상: 메인 지도 마커 조회
- 후속 작업: `2026-07-16-텍스트-마커-줌-연동-크로스페이드-design.md`

## 1. 목표

카메라 이동에 따른 공간 갱신, 새 게시글을 위한 시간 갱신, 점·말풍선 표현
갱신을 서로 분리한다.

- 점·말풍선 표현 변경은 API를 호출하지 않는다.
- 지도 이동은 새 조회 영역이 필요할 때만 API를 호출한다.
- 사용자가 지도를 움직이지 않아도 활성 지도에서 30초 이내에 새 데이터를 조회한다.
- 지도 화면이 보이지 않거나 앱이 백그라운드면 주기 조회하지 않는다.
- 네트워크 실패 시 현재 마커를 유지한다.
- 서버 `ZoomLevel`의 반경과 조회 상한이 바뀌어도 클라이언트의 고정 거리값에
  의존하지 않는다.

## 2. 현재 구조

현재 지도는 `onCameraChange`가 아니라 `onCameraIdle`에서만 자동 재조회를
판단한다. 다음 중 하나를 만족하면 500ms 뒤 `/board/map/post`를 호출한다.

- 마지막 조회 카메라 줌과 현재 줌의 차이가 1 이상
- 마지막 조회 중심과 현재 중심의 위도 또는 경도 차이가 0.0005 이상
- 텍스트 점·말풍선 모드가 변경됨

동시 조회는 `_isLoadingMarkers`와 `_pendingReloadArgs`로 합쳐 마지막 요청만
후속 실행한다. 게시글 업로드 성공, 지도 탭 재탭, 게시글 수정·삭제 후에는
기존 `refreshMap()`이 강제 새로고침을 수행한다.

현재 문제는 다음과 같다.

- 표현 변경만으로도 동일한 게시글 API를 다시 호출한다.
- 고정 0.0005도 기준은 줌 10의 50km 반경과 줌 20의 300m 반경에 동일하게
  적용되어 화면 규모를 반영하지 못한다.
- 사용자가 지도를 움직이지 않으면 새 게시글이 생겨도 갱신되지 않는다.
- HTTP 실패도 빈 목록으로 반환되어, 정상적인 0건과 구분할 수 없다.
- 실패를 빈 목록으로 처리하면 주기 갱신 도입 후 기존 마커를 모두 삭제할 수 있다.

## 3. 재조회 판단 모델

재조회 판단을 UI 상태에서 분리한 순수 정책으로 만든다. 정책은 다음 입력만
받고 재조회 이유를 반환한다.

- 현재 카메라 중심과 줌
- 마지막 성공 조회 중심과 API 줌
- 현재 화면 크기와 `meterPerDp`
- 마지막 성공 조회 시각
- 지도 탭 활성 여부와 앱 생명주기
- 카드·스택 상호작용 여부
- 현재 로딩 여부

결과는 `none`, `initial`, `apiZoomChanged`, `viewportMoved`, `stale`,
`appResumed`, `manual`처럼 이유가 드러나는 값으로 표현한다. 표현 모드 변경은
입력과 결과에 포함하지 않는다.

### 3.1 줌 변경

기존 `abs(currentZoom - lastZoom) >= 1` 대신 실제 API에 보낼 줌을 비교한다.

```text
currentApiZoom = apiZoomFor(currentZoom)
reload = currentApiZoom != lastQueriedApiZoom
```

따라서 16.5 -> 17.3처럼 차이가 1보다 작아도 서버의 `ZOOM_16`에서
`ZOOM_17`로 조회 단계가 바뀌면 재조회한다. 반대로 같은 API 줌 안의 미세한
확대·축소는 표현만 갱신하고 API를 호출하지 않는다.

### 3.2 지도 중심 이동

고정 위경도 차이 대신 화면 크기에 비례한 이동 거리를 사용한다.

```text
viewportReferenceMeters = min(viewportWidthDp, viewportHeightDp) * meterPerDp
movementThresholdMeters = viewportReferenceMeters * 0.30
reload = distance(lastTarget, currentTarget) >= movementThresholdMeters
```

`meterPerDp`는 현재 위도와 줌으로 네이버 지도 컨트롤러가 동기 계산한다.
중심 간 거리는 기존 `geolocator`의 거리 계산을 사용한다. 화면 짧은 변의 30%를
넘어간 경우만 새 영역 조회로 판단한다.

이 정책은 백엔드 반경 숫자를 클라이언트에 복제하지 않는다. 백엔드는 모든 줌에서
조회 반경이 현재 뷰포트를 덮는다는 기존 불변 조건을 유지한다.

### 3.3 카메라 정지 디바운스

공간 재조회는 계속 `onCameraIdle`에서 판단한다. `onCameraIdle` 자체가 한 이동의
마지막에 한 번 발생하므로 기존 500ms 디바운스는 300ms로 줄인다. 연속된
프로그램 카메라 이동은 기존 타이머 취소와 마지막 요청 큐가 합친다.

## 4. 시간 기반 실시간성

마지막 성공 응답으로부터 30초 뒤에 한 번 실행되는 freshness 타이머를 둔다.
`Timer.periodic`으로 고정 간격 호출하지 않고, 성공 시각을 기준으로 one-shot
타이머를 다시 예약한다.

타이머가 만료됐을 때 다음 조건을 모두 만족하면 현재 카메라 기준으로 증분
재조회한다.

- 앱이 `resumed` 상태
- 하단 네비게이션의 지도 탭이 활성 상태
- 바텀 카드와 스택 펼침이 닫힌 상태
- 다른 마커 조회가 진행 중이지 않은 상태

조건을 만족하지 않으면 `_freshnessRefreshPending`만 기록한다. 지도 탭 복귀,
앱 resume, 카드·스택 닫힘, 진행 중 조회 완료 중 먼저 발생한 시점에 pending을
소비한다. 같은 이유의 요청을 중복 큐잉하지 않는다.

지도 화면은 `IndexedStack` 안에서 계속 mounted 상태이므로 `mounted`만으로
활성 여부를 판단하지 않는다. `NavController.selectedIndex == 0`을 별도로 확인한다.

### 4.1 기존 수동 갱신

다음 기존 동작은 유지한다.

- 게시글 업로드 성공 직후 `refreshMap()`
- 현재 지도 탭 재탭 시 `refreshMap()`
- 게시글 수정·삭제 후 카드 닫기와 `refreshMap()`

이 경로는 변경된 아이콘과 캐시까지 확실히 반영하기 위한 강제 전체 갱신이다.
30초 freshness와 카메라 이동 갱신은 기존 마커를 유지하며 차이만 반영하는
`_loadMapMarkers` 경로를 사용한다.

현재 `refreshMap()`은 API 호출 전에 기존 마커와 캐시를 먼저 삭제한다. 이를
`forceRebuild` 요청으로 바꿔 기존 마커를 유지한 채 API를 먼저 호출한다. 성공
응답을 받은 경우에만 기존 오버레이와 캐시를 정리하고 전체 재구성한다. 실패하면
기존 화면을 그대로 유지한다.

## 5. 성공과 실패 구분

`BoardApiService.getMapLocationPosts`는 현재 HTTP 실패 시 빈 목록을 반환한다.
이를 성공 목록과 실패를 구분할 수 있는 반환 형태로 변경한다.

이 함수는 메인 지도 한 곳에서만 사용하므로, 이번 범위에서는 다음 계약을
사용한다.

- HTTP 200과 정상 파싱: `List<MapPost>` 반환. 빈 목록도 성공이다.
- HTTP 실패, 네트워크 예외, 파싱 실패: `null` 반환하고 오류 로그를 남긴다.

지도는 `null`을 받으면 현재 마커와 마지막 성공 조회 기준을 유지한다. 성공
목록을 받은 뒤 마커 반영까지 완료했을 때만 마지막 성공 시각과 조회 중심·줌을
갱신하고 freshness 타이머를 다시 예약한다. 강제 전체 갱신도 동일한 성공 후
교체 계약을 사용한다.

실패 후에는 15초 뒤 한 번 재시도한다. 앱이나 지도 탭이 비활성 상태면 pending으로
전환하고 활성화 시 재시도한다. 무한 즉시 재시도는 하지 않는다.

## 6. 상호작용 상태

바텀 카드나 스택이 열린 동안 자동 재조회로 `_postGroups`가 바뀌면 카드와
마커가 어긋날 수 있다. 따라서 공간·시간 자동 갱신 모두 pending으로 보류한다.

- 점·말풍선 alpha 전환은 데이터 집합을 바꾸지 않으므로 계속 허용한다.
- 카드·스택이 닫히면 현재 카메라로 정책을 다시 평가한다.
- API 줌 변경, 화면 이동, freshness가 동시에 해당되면 요청은 한 번만 보내고
  로그에는 우선순위가 높은 이유를 남긴다.
- 우선순위는 `manual > initial > appResumed > apiZoomChanged > viewportMoved > stale`이다.

## 7. 관측성

자동 조회와 스킵 로그에 이유를 명시한다.

```text
[map] reload scheduled reason=apiZoomChanged apiZoom=17 debounce=300ms
[map] reload skipped reason=withinViewport moved=82m threshold=310m
[map] freshness deferred reason=cardOpen age=34s
[map] reload failed reason=stale keepExistingMarkers=true retry=15s
```

운영 빌드에서 과도한 로그가 남지 않도록 상세 수치는 debug 모드에서만 출력한다.

## 8. 테스트

순수 재조회 정책을 단위 테스트한다.

- 텍스트 표현 모드만 변경되어도 재조회하지 않는다.
- 16.5 -> 17.3은 API 줌 변경으로 재조회한다.
- 같은 API 줌에서 화면 이동이 30% 미만이면 재조회하지 않는다.
- 같은 API 줌에서 화면 이동이 30% 이상이면 재조회한다.
- 마지막 성공 이후 30초 미만이면 freshness 조회하지 않는다.
- 30초 이상이고 지도·앱이 활성 상태면 freshness 조회한다.
- 카드, 스택, 비활성 탭, 백그라운드에서는 pending으로 보류한다.
- API 실패는 기존 마커와 마지막 성공 기준을 유지한다.
- 성공한 빈 목록은 기존 마커를 정상적으로 제거한다.
- 강제 전체 갱신 실패도 기존 마커와 캐시를 유지한다.

기존 소스 기반 클러스터 정책 테스트와 메인 지도 분석도 함께 실행한다.

## 9. 구현 순서

1. 점·말풍선 모드 변경을 API 재조회 조건에서 제거한다.
2. 고정 0.0005도 이동 기준을 화면 짧은 변 30% 기준으로 교체한다.
3. 줌 차이 1 기준을 실제 API 줌 단계 비교로 교체한다.
4. 성공·실패 반환을 분리하고 30초 freshness 및 생명주기 보류를 추가한다.
5. 이 정책을 검증한 뒤 줌 연동 크로스페이드 안전성 실험으로 이동한다.

각 단계는 자동 테스트 후 실기기에서 로그와 동작을 확인한다. 모든 작업이 끝난
뒤 변경 내용을 묶어 검토하며, 커밋은 사용자 승인 후에만 진행한다.

## 10. 범위 밖

- WebSocket, SSE, 푸시 기반 지도 동기화
- 백엔드 `ZoomLevel`이나 `/map/post` 응답 구조 변경
- 서버의 공간 쿼리 또는 인덱스 최적화
- 내 지도 화면의 재조회 정책 변경
- 텍스트 마커 크로스페이드 구현

실시간 요구가 30초보다 짧아지거나 사용자 수 증가로 폴링 비용이 문제가 될 때
증분 조회 또는 서버 푸시를 별도 설계한다.

### 11.2 실기기 체크리스트

Android와 iOS에서 다음을 확인한다.

1. 같은 줌에서 지도를 화면 30% 미만으로 움직이면 API 로그가 없다.
2. 화면 30% 이상 이동하면 idle 300ms 후 한 번만 조회한다.
3. API 줌 단계를 넘으면 한 번 조회한다.
4. 지도를 움직이지 않아도 활성 상태 30초 후 한 번 조회한다.
5. My 탭과 백그라운드에서는 30초 조회가 없다.
6. 카드·스택이 열린 동안 조회를 보류하고 닫으면 한 번 조회한다.
7. 네트워크 실패 시 기존 마커가 사라지지 않는다.

이번 계획에는 백엔드 변경과 크로스페이드 구현이 포함되지 않는다.

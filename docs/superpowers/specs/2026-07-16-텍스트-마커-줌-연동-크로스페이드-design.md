# 텍스트 마커 줌 연동 크로스페이드 설계

- 작성일: 2026-07-16
- 상태: 사용자 설계 승인, 구현 전
- 대상: `lib/screens/map/map_naver.dart`
- 관련 문서: `docs/plans/2026-07-15-마커-인터랙션-부드러움-개선.md`
- 선행 작업: `docs/superpowers/specs/2026-07-16-지도-마커-재조회-정책-design.md`

## 1. 목표

일반 텍스트 점 마커와 말풍선 텍스트 마커의 전환을 카메라 정지 이후가
아니라 줌 제스처 도중에 진행한다. 사용자가 줌 방향을 되돌리면 전환도 같은
줌 값에 맞춰 즉시 역전되어야 한다.

성공 기준은 다음과 같다.

- 줌 16.8~17.3 구간에서 점과 말풍선이 줌 진행률에 맞춰 교차 페이드된다.
- `cameraIdle`, 500ms 디바운스, API 응답 속도가 표현 전환 시점을 결정하지 않는다.
- 빠르게 줌 방향을 바꿔도 타이머 잔여 애니메이션이나 모드 점프가 없다.
- iOS에서 반복 줌과 API 재조회가 겹쳐도 프리즈가 발생하지 않는다.
- 서버의 줌 단계와 조회 상한이 바뀌어도 고정 마커 개수 가정 없이 부하를 제한한다.

이 설계는 선행 문서의 재조회 정책을 먼저 적용해 표현 전환과 API 호출을 분리한
뒤 진행한다. 크로스페이드 안전성 실험과 재조회 정책 변경을 한 번에 구현하지
않는다.

## 2. 현재 동작과 확인된 위험

현재 전환은 다음 순서에 묶여 있다.

1. 카메라가 멈춘다.
2. `onCameraIdle`이 모드 변경을 감지한다.
3. 500ms 디바운스 후 `/board/map/post`를 재조회한다.
4. 말풍선 이미지를 생성하고 기존 클러스터러블 마커의 아이콘과 크기를 교체한다.
5. 새 표현만 165ms 페이드인한다.

따라서 사용자는 줌 제스처가 끝난 뒤 한 박자 늦게 전환되는 느낌과, 기존
표현이 즉시 사라진 뒤 새 표현만 나타나는 느낌을 함께 받는다.

과거에는 이전 표현을 고스트 `NMarker`로 동적 추가해 크로스페이드를
시도했으나, iOS에서 줌 16.5 -> 17.32 전환과 재조회가 겹칠 때 거의 매번
프리즈가 재현됐다. 당시 로그는 기존 마커 재사용과 `setState` 완료 직후
멈췄으며, 고스트 `addOverlay`가 실행되는 시점과 일치했다.

`flutter_naver_map 1.4.4`의 iOS 구현은 클러스터러블 마커를 추가할 때
`clusterer.addAll` 후 `mapView`를 분리했다가 다시 연결한다. 네이버 클러스터러
내부의 동기 디스패치와 별도 오버레이 추가가 겹친 것이 유력 원인이지만,
hang sample이 없어 직접 원인으로 확정하지는 않는다.

이 설계에서는 다음을 불변 조건으로 둔다.

- 카메라 이동, 모드 전환, API 재조회 완료 직후에 단발 `addOverlay`를 호출하지 않는다.
- 전환 애니메이션을 위해 임시 또는 고스트 오버레이를 생성하지 않는다.
- 신규 점·말풍선 쌍은 하나의 `addOverlayAll` 호출에서만 등록한다.
- 안전성 실험을 통과하기 전에는 전체 조회 결과에 이중 마커를 적용하지 않는다.

## 3. 서버 조회 특성

Flutter는 `GET /board/map/post`를 호출한다. 게이트웨이의 `board` prefix 뒤에서
Board 모듈의 `GET /map/post`로 연결되며, 다음 흐름으로 조회한다.

`MapController.postList` -> `MapPostService.mapPosts` ->
`MapBoardRepositoryImpl.findLocationPosts`

저장소 쿼리는 `ST_DWithin` 반경 안의 공개 게시글을 score 내림차순으로 정렬한
뒤 `ZoomLevel.postLimit`만큼 반환한다. 현재 설정은 다음과 같다.

| 줌 | 반경 | 최대 게시글 |
|---|---:|---:|
| 10~12 | 20~50km | 30 |
| 13~14 | 5~10km | 40 |
| 15~17 | 1~3km | 50 |
| 18~20 | 300~500m | 100 |

이 값은 운영 중 변경될 수 있다. 클라이언트는 100개를 상한으로 하드코딩하지
않고, 현재 이미지 준비와 밀집 판정을 통과한 말풍선 후보 수를 기준으로 전환
품질과 호출량을 조절한다.

## 4. 선택한 구조

### 4.1 사전 등록형 마커 쌍

스택이 아닌 단일 텍스트 글마다 두 오버레이를 한 쌍으로 관리한다. 밀집 여부는
쌍을 생성하는 조건이 아니라 현재 말풍선을 노출할 수 있는지 결정하는 조건으로
분리한다. 그래야 줌이나 데이터 갱신으로 밀집 판정이 바뀌어도 새 오버레이를
추가할 필요가 없다.

- 점: 기존 `NClusterableMarker`, 게시글 ID 유지
- 말풍선: 클러스터링 대상이 아닌 `NMarker`, 별도 파생 ID 사용
- 두 마커의 위치와 탭 동작은 같다.
- 스택과 사진 글은 기존 단일 마커 구조를 유지한다.
- 밀집 지역의 단일 텍스트 글은 쌍을 유지하되 점만 노출한다.

두 오버레이는 해당 게시글을 처음 지도에 추가할 때 다른 신규 마커들과 함께
동일한 `addOverlayAll` 배치에 포함한다. 줌 전환 과정에서는 오버레이를 추가하거나
삭제하지 않고 기존 오버레이의 alpha만 변경한다.

말풍선에는 전환 시작 줌을 `minZoom`으로 지정해 낮은 줌에서 네이티브 SDK가
숨기도록 한다. 점에는 `maxZoom`을 지정하지 않는다. 높은 줌에서 바텀 카드가
열리면 기존 UX대로 말풍선을 숨기고 점을 다시 보여줘야 하기 때문이다. 평상시
전환 종료 줌 이상에서는 점 alpha 0을 유지하며, 전환 구간 밖에서는 경계 진입
시 한 번만 alpha를 확정하고 매 프레임 갱신하지 않는다.

### 4.2 줌 진행률

`NaverMap.onCameraChange`에서 `controller.nowCameraPosition.zoom`을 읽는다. 이 값은
콜백 직전에 플러그인이 갱신한 캐시이므로 네이티브 왕복이나 `await`가 없다.

전환 진행률은 다음과 같이 계산한다.

```text
raw = clamp((zoom - 16.8) / (17.3 - 16.8), 0, 1)
progress = smoothstep(raw)
dotAlpha = 1 - progress
bubbleAlpha = progress
```

진행률은 이전 상태나 타이머가 아니라 현재 줌만으로 결정한다. 따라서 줌 방향을
되돌리면 같은 줌에서 같은 alpha가 계산되고, 별도의 역방향 애니메이션이나 취소
처리가 필요 없다.

기존 16.8/17.3 히스테리시스는 표현 모드 상태가 아니라 교차 페이드 구간으로
재사용한다. API 재조회 여부 판단은 위치·정수 줌 변화만 담당하도록 분리한다.

### 4.3 이미지 준비와 캐시

점 아이콘과 말풍선 아이콘은 별도 캐시에 보관한다. 말풍선 이미지는 기존
`overlayImageFromWidget` 직렬화 경로를 반드시 사용해 iOS 임시 이미지 파일
경합 방어를 유지한다.

안전성 실험에서는 score 순으로 최대 4개의 단일 텍스트 글만 대상으로 삼고,
두 이미지가 준비된 뒤 한 배치로 등록한다. 전체 적용 단계에서는 이 안전성
상한을 제거하고 다음 원칙을 적용한다.

- 단일 텍스트 글의 말풍선 오버레이는 공유 투명 아이콘으로 먼저 같은 배치에
  등록하고, 실제 이미지가 준비되면 이미 등록된 일반 `NMarker`의
  아이콘만 교체한다.
- 이미 생성한 말풍선 이미지는 게시글 ID와 표시 내용 기준으로 재사용한다.
- 무제한 `Future.wait`로 위젯 이미지를 동시에 생성하지 않는다.
- 말풍선 준비가 늦으면 점을 유지하고, 준비되지 않은 말풍선을 전환 대상으로
  포함하지 않는다.
- 이미지가 준비됐다는 이유로 전환 도중 새 오버레이를 추가하지 않는다.

## 5. 성능 제어

`setAlpha`는 마커마다 플랫폼 채널 호출을 발생시키므로 카메라 콜백마다 모든
후보를 무조건 갱신하지 않는다.

- 전환 구간 안에서만 alpha를 갱신한다.
- 진행률을 후보 수에 따라 정한 버킷으로 양자화한다.
- 직전 버킷과 같으면 아무 호출도 하지 않는다.
- 품질 기준은 서버 `postLimit`이 아니라 현재 전환 후보 수로 계산한다.
- 최대 갱신 빈도는 30Hz로 제한한다.
- 초기 플랫폼 호출 예산은 한 방향 전체 줌 통과당 600회로 둔다.
- 후보 수를 `N`이라 할 때 버킷 수는 `min(12, 600 ~/ (2 * N))`으로 계산한다.
- 계산된 버킷 수가 3보다 작으면 이중 마커 alpha 갱신을 사용하지 않고 폴백한다.
- 후보가 호출 예산을 초과하면 프리즈나 프레임 드롭을 감수하지 않고 단일 마커
  폴백 또는 네이티브 줌 경계 전환으로 낮춘다.

호출 횟수, 후보 수, 버킷 변경 횟수는 debug 로그로 측정한다. 600회는 초기 안전
예산이며 실기기 측정으로 낮출 수 있다. 고정된 20/50/100개 구간은 정책으로
두지 않는다.

## 6. 카드 선택과 기존 상태 연동

바텀 카드가 열려 있을 때 말풍선을 점으로 억제하는 기존 UX는 유지한다.

- 카드 열림: 말풍선 alpha를 0, 점 alpha를 1로 강제한다.
- 카드 닫힘: 현재 줌 진행률을 다시 적용한다.
- 선택 강조, 캡션, z-index는 점 마커가 소유한다.
- 말풍선은 카드 비트맵 자체에 제목과 본문이 있으므로 별도 캡션을 사용하지 않는다.
- 점과 말풍선의 탭 리스너는 같은 게시글 선택 함수를 호출한다.

재조회로 게시글이 사라질 때는 점과 말풍선을 함께 제거하고 양쪽 캐시와 참조를
같이 정리한다. 한쪽만 남는 상태를 허용하지 않는다.

## 7. 실패 처리와 폴백

다음 조건 중 하나라도 발생하면 이중 마커 전체 적용을 중단한다.

- iOS 안전성 실험에서 프리즈 또는 무한 대기가 한 번이라도 발생한다.
- 전환 구간에서 지속적인 프레임 드롭이 관찰된다.
- 점·말풍선 중 한쪽 오버레이만 남거나 탭이 유실된다.
- API 재조회와 줌 전환이 겹칠 때 네이티브 `overlay can't found`가 반복된다.

폴백은 신규 오버레이가 필요 없는 단일 마커 방식이다.

1. 줌 진행률 전반부에 현재 표현을 페이드아웃한다.
2. alpha가 0인 중앙 지점에서 미리 준비한 아이콘·크기·캡션을 교체한다.
3. 진행률 후반부에 새 표현을 페이드인한다.

이 방식은 진짜 크로스페이드는 아니지만, 과거 고스트 방식과 달리 오버레이를
추가하지 않으며 줌 방향 반전도 현재 진행률로 계산할 수 있다.

## 8. 검증 단계

### 8.1 안전성 실험

먼저 score 순 최대 4개의 전환 후보에만 사전 등록형 이중 마커를 적용하고,
로그에 안전성 상한 적용 여부와 실제 대상 ID를 남긴다.

- iOS 실기기에서 줌 16.5 <-> 17.5를 30회 이상 반복한다.
- 전환 도중 API 재조회가 발생하도록 위치와 줌을 함께 바꾼다.
- 전환 구간에서 줌 방향을 빠르게 반복 반전한다.
- 바텀 카드를 열고 닫으며 말풍선 억제와 복원을 반복한다.
- 신규 마커 추가, 기존 마커 재사용, 삭제가 각각 포함되도록 테스트한다.

실험 로그에는 API 시작/완료, 배치 추가 시작/완료, 전환 후보 수, 진행률 버킷,
오버레이 삭제를 남긴다. 프리즈가 발생하면 Xcode에서 hang sample을 확보해
추정이 아닌 실제 대기 스택을 확인한다.

### 8.2 전체 적용 검증

안전성 실험을 통과한 뒤 제한을 제거하고 다음을 확인한다.

- 후보 1개, 20개 안팎, 현재 서버 최대치에 가까운 데이터에서 프레임 안정성
- 줌인/줌아웃 양방향의 동일한 시각 결과
- API가 느리거나 실패해도 기존 마커의 줌 전환이 계속 동작하는지
- 스택·사진 마커가 이중화되지 않고, 밀집 텍스트의 말풍선이 노출되지 않는지
- 카드 선택, 캡션, 클러스터 탭, 스택 펼침·접힘 회귀 여부
- Android와 iOS 모두에서 `overlay can't found`, 프리즈, 탭 유실이 없는지

순수 진행률 계산과 호출 예산 계산은 단위 테스트로 고정한다. 네이티브 오버레이
수명주기와 프리즈 여부는 자동 테스트로 충분히 검증할 수 없으므로 실기기 통과를
배포 조건으로 둔다.

## 9. 범위 밖

- `flutter_naver_map` 포크 또는 네이티브 배치 alpha API 추가
- 사진 마커와 스택 마커의 이중 표현
- 전환 구간이나 말풍선 디자인 자체 변경
- 백엔드 `ZoomLevel`, 조회 반경, `postLimit` 변경
- 내 지도 화면의 동일 기능 적용

메인 지도에서 안전성과 체감을 확인한 뒤 내 지도 적용 여부를 별도 작업으로
결정한다.

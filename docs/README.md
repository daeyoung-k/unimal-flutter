# 📚 unimal-flutter 문서

스토맵 앱의 모든 설계·계획·할일·트러블슈팅 문서는 **이 `docs/` 아래에만** 둔다.
`lib/` 등 코드 폴더 안에 설계 문서를 만들지 않는다.

## 폴더 구조

| 폴더 | 용도 | 무엇을 담나 |
| --- | --- | --- |
| `architecture/` | 앱 구조·방향성 | 화면 구조, 상태관리, 디자인 토큰 방침, 장기 방향성. 살아있는 문서 |
| `specs/` | 기능별 설계 | "무엇을 / 왜" — 화면 요구사항, UI/UX 설계, 동작 정의 |
| `plans/` | 구현 계획 | "어떻게 / 어떤 단계로" — 작업 분해, 순서, 체크리스트 |
| `todo/` | 할일·백로그 | 아직 착수 안 한 아이디어/작업 메모 |
| `troubleshooting/` | 디버깅·장애 기록 | 문제 원인과 해결 과정. 같은 삽질 반복 방지 |

> `release-note-format.md`처럼 어느 카테고리에도 안 맞는 표준/포맷 문서는 `docs/` 루트에 둔다.

## 명명 규칙

- `specs/`, `plans/`, `troubleshooting/` → **`YYYY-MM-DD-주제.md`** (작성일 기준)
  - 예: `2026-06-14-mypage-redesign.md`
- `architecture/`, `todo/` → **주제 기반** (날짜 없음, 살아있는 문서)
  - 예: `스토맵-방향성-구조-설계.md`, `map-tab-discovery-layer.md`
- 한글 파일명은 공백 대신 하이픈(`-`) 사용.

## 작성 규칙

- 한 화면/기능은 보통 `specs`(설계) → `plans`(구현 계획) 한 쌍으로 남긴다. 같은 주제면 같은 날짜·주제명을 맞춘다.
- 자동화 도구(superpowers 등)가 `docs/superpowers/...`에 문서를 만들면, 위 카테고리 폴더로 옮기고 `superpowers/`는 비워둔다.
- 완료된 작업의 plan/spec은 지우지 말고 둔다 — 의사결정 기록이자 포트폴리오 자산이다.
